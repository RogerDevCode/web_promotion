# ÍNDICE DE ESPECIFICACIÓN FUNCIONAL DEL CATÁLOGO

## 1. Objetivo

Esta carpeta define el comportamiento funcional del contexto de catálogo
de VoiceShop.

El catálogo es la fuente oficial para:

- productos;
- categorías;
- marcas;
- variantes;
- presentaciones;
- atributos;
- imágenes;
- descripciones;
- precios de referencia cuando el diseño lo asigne a este contexto;
- aliases;
- disponibilidad comercial del producto;
- vigencia;
- búsqueda;
- navegación.

El catálogo no es inventario.

Que un producto exista en catálogo no significa que tenga stock.

El catálogo no procesa pagos ni pedidos.

## 2. Alcance

Incluye:

- búsqueda;
- consulta de producto;
- categorías;
- marcas;
- filtros;
- ordenamiento;
- paginación;
- aliases;
- resolución de referencias;
- productos activos e inactivos;
- variantes;
- presentaciones;
- datos visibles;
- imágenes;
- caché funcional;
- versionado;
- errores;
- observabilidad;
- QA.

No incluye:

- reservas;
- stock;
- carrito;
- pedido;
- pago;
- entrega;
- promociones complejas;
- administración detallada del catálogo;
- persistencia técnica concreta.

## 9. Reglas transversales

RULE-CAT-001

Todo producto pertenece a un tenant.

RULE-CAT-002

Todo producto posee Product ID.

RULE-CAT-003

Todo producto visible debe estar ACTIVE o permitido explícitamente.

RULE-CAT-004

El catálogo no afirma stock.

RULE-CAT-005

El catálogo no acepta IDs inventados por una LLM.

RULE-CAT-006

Toda respuesta debe indicar versión o vigencia cuando corresponda.

RULE-CAT-007

La búsqueda debe ser determinística para mismos datos y parámetros.

RULE-CAT-008

Los resultados deben respetar aislamiento de tenant.

RULE-CAT-009

Las imágenes deben ser referencias autorizadas.

RULE-CAT-010

Los aliases deben estar versionados.

RULE-CAT-011

Una referencia ambigua no debe resolverse arbitrariamente.

RULE-CAT-012

Los filtros deben validarse.

RULE-CAT-013

El ordenamiento debe usar campos permitidos.

RULE-CAT-014

La paginación debe ser estable.

RULE-CAT-015

Los productos inactivos no se agregan a nuevos carritos.

## 10. Fuente de verdad

Puede existir:

- catálogo interno;
- ERP;
- PIM;
- servicio externo;
- catálogo híbrido.

VoiceShop debe consumir una interfaz oficial.

La LLM no es fuente de catálogo.

## 11. Datos oficiales y derivados

OFFICIAL

- Product ID;
- SKU;
- nombre;
- categoría;
- marca;
- presentación;
- status.

DERIVED

- search score;
- display label;
- normalized search terms;
- aliases sugeridos;
- ranking.

Los derivados no deben cambiar hechos.

## 12. Relación con precios

La arquitectura puede asignar el precio a:

- Catalog Context;
- Pricing Context;
- Promotion Context.

Cuando el catálogo entregue precio, debe indicar:

- amount;
- currency;
- price type;
- version;
- validity;
- tax policy.

## 13. Relación con inventario

Catalog:

"El producto existe."

Inventory:

"Hay unidades disponibles."

Nunca inferir stock por presencia en catálogo.

## 14. Relación con conversación

El motor conversacional usa catálogo para:

- resolver productos;
- generar listas;
- aclarar;
- validar IDs;
- obtener nombres oficiales;
- mostrar detalles.

## 15. Relación con carrito

Antes de agregar:

- Product ID existe;
- Product status permite;
- presentación válida;
- precio oficial;
- inventario cuando aplique;
- tenant coincide.

## 16. Listas

Toda lista debe poseer:

- List ID;
- versión;
- Search Request ID;
- posiciones;
- Product IDs;
- expiración;
- tenant;
- filtros.

Esto permite resolver:

- "la segunda";
- "esa";
- "la más barata".

- rate limit;

- no URLs arbitrarias.
