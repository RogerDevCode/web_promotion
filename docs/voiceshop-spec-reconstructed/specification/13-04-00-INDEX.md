# ÍNDICE DE ESPECIFICACIÓN FUNCIONAL DE INVENTARIO

## 1. Objetivo

Esta carpeta define el comportamiento funcional del contexto de
inventario de VoiceShop.

El inventario es la fuente oficial para responder:

- si existe stock;
- cuántas unidades están disponibles;
- en qué sucursal;
- qué unidades están reservadas;
- qué unidades están comprometidas;
- qué unidades están bloqueadas;
- cuál es la vigencia de una lectura;
- cómo se evita vender más de lo disponible;
- cómo se gestionan reservas;
- cómo se liberan reservas;
- cómo se concilian diferencias.

El inventario no define la identidad del producto.

El catálogo define Product ID, nombre, marca y presentación.

El inventario no procesa pagos.

## 5. Entidades y agregados

Inventory Balance

Representa cantidades de un Product ID en una ubicación.

Stock Reservation

Representa una retención temporal de unidades.

Inventory Adjustment

Representa un cambio autorizado.

Inventory Snapshot

Representa una lectura versionada.

Location

Sucursal, bodega u otra ubicación.

## 7. Tipos de cantidad

ON_HAND

Existencia física registrada.

AVAILABLE

Cantidad utilizable para nuevas operaciones.

RESERVED

Cantidad temporalmente retenida.

COMMITTED

Cantidad asociada a pedido confirmado.

BLOCKED

Cantidad no utilizable.

DAMAGED

Cantidad dañada.

IN_TRANSIT

Cantidad en tránsito.

SAFETY_STOCK

Cantidad protegida por política.

La fórmula exacta de AVAILABLE debe estar documentada.

## 8. Fórmula funcional inicial

Ejemplo conceptual:

AVAILABLE =
ON_HAND
- RESERVED
- COMMITTED
- BLOCKED
- SAFETY_STOCK

La fórmula final depende de las reglas del negocio.

No debe calcularse en una LLM.

## 12. Principios

RULE-INV-001

Todo balance pertenece a tenant, location y Product ID.

RULE-INV-002

Todo Product ID debe validarse contra catálogo.

RULE-INV-003

El inventario es la fuente oficial de stock.

RULE-INV-004

La LLM no calcula ni modifica stock.

RULE-INV-005

Toda escritura usa idempotencia.

RULE-INV-006

Toda escritura usa versión o control equivalente.

RULE-INV-007

No se permite stock disponible negativo salvo política explícita.

RULE-INV-008

Las reservas expiran.

RULE-INV-009

Las reservas deben liberarse de forma idempotente.

RULE-INV-010

Una reserva no equivale a un pedido confirmado.

RULE-INV-011

Un producto activo en catálogo puede tener stock cero.

RULE-INV-012

Un resultado UNKNOWN no se reintenta a ciegas.

RULE-INV-013

Toda cantidad debe ser entera o usar unidad formal permitida.

RULE-INV-014

Toda transición debe emitir evento.

RULE-INV-015

Toda operación debe ser auditable.

## 13. Relación con catálogo

Antes de consultar o modificar:

- Product ID existe;
- Product ID pertenece al tenant;
- presentación compatible;
- status permitido.

El inventario no corrige datos de catálogo.

## 14. Relación con carrito

El carrito puede:

- consultar disponibilidad;
- solicitar reserva;
- modificar reserva;
- liberar reserva.

La política debe definir si el carrito reserva inmediatamente o sólo al
confirmar.

## 15. Relación con pedido

El pedido puede:

- convertir reserva a committed;
- liberar en cancelación;
- conciliar UNKNOWN;
- registrar consumo.

## 16. Relación con pagos

El inventario no interpreta pago.

Puede reaccionar a eventos oficiales:

- PaymentApproved;
- PaymentFailed;
- OrderCancelled.

## 17. Relación con voz

La voz puede consultar:

"¿Quedan seis unidades?"

La respuesta debe usar Inventory Result oficial.

Para reservar, la voz sigue:

- intención;
- slots;
- Tool Proposal;
- autorización;
- Command;
- Inventory Handler.

## 18. Consistencia

Opciones:

STRONG

Lectura y escritura fuertemente consistentes.

EVENTUAL

Lecturas pueden retrasarse.

HYBRID

Reservas fuertes, lecturas generales eventuales.

La política debe documentarse por operación.

## 19. Versionado

Inventory Version debe incrementarse ante cambios.

Puede ser:

- por balance;
- por Product + Location;
- por aggregate;
- por snapshot.

## 20. Disponibilidad y vigencia

Toda respuesta debe incluir:

- available quantity;
- measured_at;
- Inventory Version;
- source;
- freshness;
- location;
- policy.

## 21. Reservas

Una reserva debe incluir:

- Reservation ID;
- owner;
- quantity;
- location;
- product;
- status;
- expiration;
- idempotency;
- versions.

## 22. Reservas parciales

Debe definirse:

ALL_OR_NOTHING

Si no existe cantidad total, no reservar.

ALLOW_PARTIAL

Reservar parte y reportar.

Por defecto recomendado para compra minorista:

ALL_OR_NOTHING, salvo regla explícita.

## 23. Expiración

Toda reserva temporal debe poseer expires_at.

Un worker o proceso debe:

- detectar;
- liberar;
- emitir evento;
- manejar duplicados;
- conciliar.

## 24. Ajustes

Sólo actores autorizados pueden:

- aumentar;
- disminuir;
- bloquear;
- desbloquear;
- marcar dañado;
- conciliar.

Todo ajuste requiere razón.
